Chapter 3: Object Oriented Programming: Classes and Objects

Introduction

Imperative programming structured in the form of functions and procedures allows to split a problem into several sub-problems. Each method solves a part of the problem. Consequence: the architecture of the program is made from the treatments it performs

Disadvantages of imperative programming:

Imperative programming appears limited when :

  • we want to architect the application: gather processes, create new ones, delete them. How to gather these processes according to their semantics?
  • to maintain / evolve the code.
  • we want to make sure that the user does not modify some "critical" variables of the code: problem of code robustness

Object Oriented Programming (OOP) was developed as early as 1960 (by Dah and Nygaard) and then in 1970 by Alan Kay by proposing a set of principles to limit these problems. It is a programming paradigm

The OOP allows to model the pb from the data they process and how these data interact with each other. These data are called Object. The architecture of the program is made from the data it processes.

OOP Objectives:

  • Facilitate the reuse of code
  • Facilitate its evolution
  • Improve the design of the solution in order to improve its maintenance

Example : Modeling of the functioning of a TV

  • Objects : - Screen (size, resolution, ...)
  • Remote control (color, weight)
  • Methods : - turn on / turn off
  • increase / decrease the sound
  • change channel

History :

  • 1967: SIMULA -> origin of OOP. Modeling of simulations. Natural description by objects that change (OOP + event-driven programming)
  • 70's : SmalTalk -> Development of OOP. First All Object language. Advent of WIMP windowed environments (Window, Menu, Pointing Device)
  • 80's : Appearance of OO languages, C++, Object Pascal, Delphi, Objective C
  • 1996 : Appearance of the Java language : "high-level" OO language
  • 1998: C++ standard

An object is the semantic representation of a real object

A person, a car, a TV, ...

Fondamental

An object is defined by :

  • A name : the name of the object
  • One or more states (color = blue, weight = 10kg,...)
  • One or more behaviors (treatments)

Exemple

A black Peugeot 207 driving at 90km/h

  • Name: Peugeot 207
  • Status: color = black, speed = 90km/h
  • Behavior: drives

They can also return the value of a state of the object. For example, the treatment giveSpeed() can return the value of the car's speed.

  • The treatments of an object can modify the characteristics of the object. For example, the changeColor()processing can change the color of the car.
  • They can also return the value of a state of the object. For example, the treatment giveSpeed()can return the value of the car's speed.

Methods of a class

Introduction

Each class must have 4 types of methods:

  • methods called constructors allowing to build objects ;
  • methods allowing to return the value of each attribute. These methods are called getters;
  • methods allowing to return the value of each attribute. These methods are called setters;
  • methods allowing to change the state of the attributes (for example drive, stop, ...)

Constructor of a class

Constructeur d'une classe

A constructor is a method that allows to build objects of the same type as the class. In Python a constructor is declared in the form : 

   1
def __init__(self, param1 = valueDefault1, param2 = valueDefault2, ... paramN = valueDefaultN)
  • self is the object we create (the instance we manipulate) ;
  • param1, .. paramN : the parameters of the constructor. Generally, these parameters are the variables that initialize the values of the attributes. It is possible to give default values (valueDefault1, ..., valueDefaultN)

* Attention

Instantiation and manipulation of objects

Introduction

An object is constructed from a class by calling a constructor. We say that an object is an instantiation of a class

Instantiation

To construct an object, the class constructor must be called implicitly with the following syntax: 

   1
myObject = ClassName(valueParam1, valueParam2, ..., valueParamN)

Construction of a black car of speed 10km/h : Exemple 

   1
car1 = Car("black",20.0)

If the class constructor has parameters initialized with default values, it is possible to initialize the value of the attributes with these default values: 

   1
car2 = Car()

In this case, the car created will be red and its speed will be 0 km/h.

Manipulate an object

The manipulation of an object consists of either:

  • to access or change the value of an attribute ;
  • to call a processing method (for example accelerate, decelerate, ...).

The value of an attribute is accessed by specifying the attribute by its name and using the following syntax: 

   1
objectName.attributeName

It is possible to change the value of an attribute by replacing it (using the assignment operator) with a new value. This new value must be of the same type as the attribute. 

   1
objectName.attributeName = newValue

A processing method is accessed by specifying the method name (with parameter values, if any) and using the following syntax: 

   1
objectName.method(valueParam1, valueParam2, .., valueParamN)

Composition and Aggregation

Introduction

The attributes of a class can be objects from other classes. A class can thus group together objects created from another class.

In other words, an attribute can refer to an object. Depending on the life cycle of this object, we will speak either of aggregation or composition.

The composition

The composition translates a relation of subordination, of belonging. It is a strong association that represents a subordination link. Semantically, it expresses the relation "owns".

A class B is aggregated in a class A if the life cycle of an instance of B depends on that of A.

For example: A car has an engine. If we destroy the car then we destroy the engine.

At the code level, this is translated by the presence of an instance of class B contained in the containing class A. The instance of class B is created by class A.

Car and Engine Class : Exemple

The aggregation

Fondamental

In the case of aggregation, there is no notion of container/content, just a relationship used. The two instances know each other at a given time. Their life cycle is not linked and they are independent.

At the code level, the aggregation results in the presence of an instance of class B contained in the containing class A. This instance is not created by the class A
